# L3 评估层规范 v0.1
## RevUnsup项目 — 工业缺陷检测黑盒优化场景

> 目的：让硕士生知道要做什么；让Agent知道自动评价的边界；让博士生(L2编排者)放手不再操心实现细节。
>
> 写作日期：2026-04-09
> 状态：草案v0.1，待讨论

---

## 0. 这份文档解决什么问题

RevUnsup项目当前卡在【不知道下一步做什么】。原因是L2编排层缺少"问题理解"模块——而问题理解的前提是【有一套可信的评价体系，能告诉我们当前在哪里、距离目标多远】。

L3评估层就是这套评价体系。它把"目标:提精度+压速度"翻译成可量化、可复现、可监督的评价动作。

---

## 1. 评价对象（四个维度）

```
L3评估层
├── ① 数据集评价      ← 输入端
├── ② 模型评价        ← 算法/.exe端
├── ③ 打包文件评价    ← 工程端
└── ④ 评价方式本身    ← 元层
```

---

## 2. 数据集评价

### 2.1 评什么
| 子项 | 含义 | 判定方式 |
|---|---|---|
| **质量** | 标注是否准确、图像是否清晰、是否有重复 | 抽样人工检查 + 自动去重 |
| **数量** | 是否够支撑统计意义的精度对比 | 最低50张，分类别覆盖 |
| **丰富度** | 是否覆盖典型场景 / 边界 / 难例 | 按场景分桶，每桶≥10张 |

### 2.2 不达标怎么办
- **数量不足** → 优先补难例和边界case，不盲目堆数
- **质量不足** → 标注复审，错标剔除
- **丰富度不足** → 分桶后定向补齐缺失桶

### 2.3 当前状态
- 60%已整理，剩余40%交人工补充（已有计划）
- 输出：`test_set/` 目录 + `dataset_meta.json`(每张图的桶/标签/来源)

---

## 3. 模型评价

### 3.1 静态评价（不跑也能评）
| 子项 | 含义 | 怎么测 |
|---|---|---|
| **健壮性** | 异常输入下是否崩溃 | 喂空图/超大图/非图像文件，看exe行为 |
| **完整性** | 声称的功能是否都能调用 | 接口探测脚本逐项调用 |

### 3.2 动态评价（跑起来才能评）
| 子项 | 含义 | 指标 |
|---|---|---|
| **耗时** | 推理快慢 | 单张推理时间ms / 100张平均ms / 吞吐fps |
| **质量** | 精度好坏 | mAP / Precision / Recall / F1 (按任务定) |
| **输入要求** | 对输入的约束 | 支持的分辨率/格式/通道数/数值范围 |

### 3.3 baseline 协议
**所有改进必须以baseline为对照**。baseline跑一次即可，记录到 `baseline.json`：
```json
{
  "exe_version": "...",
  "test_set_version": "v0.6",
  "metrics": {
    "mAP": 0.xx,
    "avg_inference_ms": yy,
    "p95_inference_ms": zz
  },
  "failure_cases": ["img_001.jpg", "img_023.jpg", ...],
  "timestamp": "2026-04-09T..."
}
```

## 4. 打包文件评价

### 4.1 可复用性
- 能否被Harness框架以Worker形式直接引入？
- 接口是否标准化（输入/输出/错误码）？
- 依赖是否可声明（requirements / Dockerfile）？

### 4.2 格式规范性
- 目录结构是否清晰（model/ data/ scripts/ docs/）
- 命名是否一致
- 是否有README说明如何调用

### 4.3 当前状态
.exe是黑盒，无法评内部结构。只能从【外部接口】角度评价：
- 输入：能接受什么格式的图片？
- 输出：返回什么结构？
- 调用方式：命令行参数？stdin/stdout？文件读写？

这部分需要先做一次【接口探测】，产出 `exe_interface.md`。

## 5. 评价方式（怎么评）

### 5.1 机械评价（Agent 自动）
**适用场景**：可量化、可复现、不需要主观判断的评价。
- 跑测试集 → 出metric
- 接口探测 → 出能力清单
- 边界测试 → 出健壮性报告
- 性能测试 → 出耗时分布

**输出位置**：`evaluations/<日期>/<实验ID>.json`

### 5.2 人工监督（硕士生）
**适用场景**：需要主观判断、异常识别、数据完整性核查。

硕士生的职责（**中间版边界**，2026-04-09 决策）：

**基础职责（每日/每周固定）**：
- ✅ 每天看Dashboard面板
- ✅ 检查L4状态盘的数据完整性
- ✅ 标记异常case
- ✅ 写周报汇总

**外围技术活（按需分配，不重要的活）**：
- ✅ 数据预处理脚本
- ✅ 测试集补充标注
- ✅ 实验日志整理
- ✅ 简单的辅助脚本

**禁区**：
- ❌ 不参与算法决策
- ❌ 不动核心代码/框架

**定位**：状态守门人 + 外围辅助。不是算法工程师，但可以做明确边界的小活。

### 5.3 面板可视化
所有评价结果必须落到面板上，硕士生才能看。面板的卡片结构（对应4维评价）：
```
[数据集卡]   [模型卡]   [打包文件卡]   [评价记录卡]
   质量          静态         接口             今日实验
   数量          动态         可复用性         当前最优
   丰富度        baseline     格式             异常标记
```

## 6. 角色职责矩阵

| 角色 | 是什么 | 做什么 | 不做什么 |
|---|---|---|---|
| **博士生(L2编排)** | 大脑 | 看面板→决策下一步杠杆 | 不亲自整理数据/不写脚本 |
| **硕士生(L4监督)** | 状态守门人 | 看面板/标异常/周报 | 不参与算法决策 |
| **Agent(L1执行+L3机械)** | 手+评委 | 跑实验/出指标/进数据库 | 不替代人工最终判断 |

### 6.1 任务分配优先级（劳动力协议）

任何新任务出现时，按以下顺序找承担者：

```
① Agent (默认首选)
   ↓ Agent能做就让Agent做
   ↓ Agent做不了 / 失败 / 超出能力
② 硕士生
   ↓ 硕士生能做就让硕士生做
   ↓ 硕士生也做不了 / 没空 / 需要专业判断
③ 博士生(用户自己)  ← 兜底
```

**核心原则**：博士生(L2)的时间是最稀缺资源。所有任务默认都不应该由博士生亲自做，除非Agent和硕士生都搞不定。

**适用场景示例**：
- exe接口探测 → ① Agent写探测脚本
- 数据集补充标注 → ② 硕士生
- 决定下一个杠杆试什么 → ③ 博士生（这是L2编排，无法外包）
- 跑测试集 → ① Agent
- 异常case人工核查 → ② 硕士生
- 定义"提升"的标准 → ③ 博士生（这是评价体系设计）

**不可外包的事**（必须博士生做）：
- 战略决策（论文方向、实验设计）
- 评价体系本身的设计（什么算"提升"）
- 编排逻辑（下一步做什么）

## 11. 已知问题 / 需要注意的事项

### 11.1 源码加密 + 白名单读取限制 ⚠️ 高优先级 / 待研究

**问题描述**：
甲方提供的源码使用了某种加密/保护机制。当前观察：
- 用户用【白名单内的特定软件】打开能看到代码内容
- 用户在文件管理器里能列出文件夹
- 但 Agent (Claude/Codex/OpenCode) 通过常规 `Read` 工具能否读取，**未测试，未知**

**可能的影响**：
- 如果 Agent 读不到 → 整个"Agent 自动改源码"路径塌方
- 任务分配优先级里"① Agent 默认首选"在源码相关任务上失效，必须降级到 ② 硕士生 或 ③ 博士生
- L3 §3 模型评价(静态评价部分)难以自动化

**待研究的问题**：
1. 加密原理是什么？(EFS/DRM/IDE插件层/文件签名校验/...)
2. 白名单是基于进程名、签名、路径还是别的？
3. 普通 `cat`/`open()` 系统调用能读到什么？(密文?乱码?权限拒绝?)

**应对方案（按可行性排序）**：
- **A 找出加密机制 → 申请白名单授权** ← 最干净，但耗时
- **B 用户白名单软件打开后复制粘贴给 Agent** ← 短期可行，但失去自动化
- **C 截图 + 视觉模型识别** ← 麻烦，精度有限
- **D 把代码改写工作完全交给博士生/硕士生在白名单软件内做** ← Agent 退化为评估角色
- **E 申请甲方提供未加密版本** ← 取决于关系

**当前状态**（2026-04-09）：
- 待用户给出测试链接 → Agent 实测能否读取
- 测试结果决定走哪个应对方案
- 在测试结果出来前，所有"Agent 改源码"的任务暂缓
