## 项目目标 (一句话)

把【本地已实现的算法】用五层架构封装成【数据→训练→推理→验收】完整流程, 核心目标: **解决推理时间过长的问题**。

---

## 五层架构 (Napoleon 定义)

```
┌──────────────────────────────────────────────┐
│ L5 开发者协议 + 操作封装                      │  ← 最底层的"手"
│   定义协议 + Agent 操作封装                   │
├──────────────────────────────────────────────┤
│ L4 通讯层                                    │  ← Agent 之间说话
│   组件间通信, 各种对接协议(A2A 交接等)       │
├──────────────────────────────────────────────┤
│ L3 评估层                                    │  ← 方案靠不靠谱
│   确保方案可靠性 + 检查代码问题               │
├──────────────────────────────────────────────┤
│ L2 Master/主脑                               │  ← 整合 + 决策
│   整合 Worker 输出 + 找最终方案 + 多方案择优 │
├──────────────────────────────────────────────┤
│ L1 Worker 层 (3 个 Agent)                     │  ← 真正干活
│   解决现实问题(推理时间过长)                  │
│   性能调优: 定位耗时 → 参考本地经验 → 优化   │
│   任务目标: 找耗时环节 + 针对性处理           │
└──────────────────────────────────────────────┘
```

## 根本症结: 为什么"互相等待"

**没人真正跑过任何代码**, 所以:
- L1 没数据 → L3 无评估对象 → L2 无决策依据 → L4 无消息可传 → L5 无操作可封装

**唯一的解药**: 让 L1 真的跑起来, 哪怕只跑出【一个时间数字】。

> "你不跑起来的话, 根本不知道时间到底耗在什么地方" — 羊爸爸

OldMajor_Codex 在 OM-001 里给出了 5600→83.3 ms 的数字链, 但那是【旧版 selftest 6 张图】的方法论证明, 不是【鑫业城业务数据】的 baseline。
