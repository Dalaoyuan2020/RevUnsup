## 3. 有效发现：哪些数字/结论值得记住

### 发现 1：KNN 确实是最大瓶颈
- 证据最硬。
- 历史数字：
  - 单线程总推理 ~5700 ms
  - OpenMP 总推理 1427 ms
  - FAISS IVFFlat 总推理 83.3 ms
- 这说明当时的人已经明确看到“不是模型前向慢，主要是 KNN 搜索慢”。

### 发现 2：真实部署阈值不能直接迷信训练产物
- `milestone1/thresholds.json` 里 `raw_pixel_threshold=NaN`
- `milestone3/calibration_results.json` 又重新扫 acceptance 集，把 `raw_image_threshold` 改成 19.1151
- 意味着：部署侧必须有重新校准这一步，不然阈值容易漂。

### 发现 3：PatchCore 的关键细节不是“随便抄”
- M2 明确记了 3 个重要点：
  - 用 `AvgPool2d(3,1,1)`
  - 不做 L2 归一化
  - 归一化公式不是普通 min-max，而是 `((raw - threshold) / (max - min)) + 0.5`
- 这些都不是大方向，但一错就会导致分数方向或数值整体不对。

### 发现 4：公开数据能证明流程可行，但不能证明业务稳了
- MVTec transistor AUROC=0.96，很好看。
- 但 acceptance 集上 Good/Bad 分数仍重叠，C++ 校准后依然有误报漏报。
- 说明公开数据只能做流程验证，不能替代业务验证。

### 发现 5：当前这套经验更像“先打通，再收敛”
- 证据包括：
  - selftest 只有 6 张图
  - 训练时间 2.279 秒
  - 多份交接文档反复强调“这是流程打通，不代表最终业务精度”

## 4. 失败记录：哪些路不通

### 失败 1：直接从 checkpoint 抽像素阈值，不可靠
- 证据：
  - `ISSUE_FEEDBACK_TO_Codex.md` 明说 `pixel_threshold=NaN`
  - 还会让 JSON 不合法，阻塞下游
- 后果：
  - Agent 2 一度被卡住
  - 最后只能用 fallback 0.5
- 教训：
  - 阈值抽取脚本必须对坏值兜底

### 失败 2：ONNX 推理数值想一把对齐 Python，不现实
- 证据：
  - M2 报告：AUROC 目标 0.90，实际 0.82
  - 原因被归到预处理差异，例如 `cv2.resize vs PIL resize`
- 教训：
  - 先保证方向对，再做数值贴合
  - 不要把“完全一致”当成第一目标

### 失败 3：Good/Bad 可分性没有想象中干净
- 证据：
  - `calibration_results.json` 里 good max=21.4175，bad min=16.8127，完全重叠
  - `RECEIPT.md` 里也直接写了 Good/Bad 分数范围重叠
- 教训：
  - 单一图像阈值不是银弹
  - 校准只能缓解，不能消灭数据分布差异

### 失败 4：FAISS-GPU 在这台 Windows 机器上不值得硬冲
- 证据：
  - conda 超时
  - pip 无 Windows wheel
  - GitHub 预编译下载也失败
- 教训：
  - 在这台机器上继续追 FAISS-GPU，大概率是时间黑洞

### 失败 5：环境本身会反复咬人
- 证据：
  - `TRANSFER_FROM_Codex_TO_ClaudeCode.md` 里有 PyTorch DLL 初始化失败
  - `PROJECT_REVIEW.md` 里也提到 Windows 环境依赖问题
  - `TRANSFER_GPU_OPTIMIZATION_STATUS.md` 又记录了 onnxruntime 版本问题，1.24.4 失败，1.18.0 成功
- 教训：
  - 环境固定比“装最新版”更重要

## 6. 与 RU-001 三个金矿的对应

### 金矿 1：modelInputW/H 硬编码可绕过
- 旧版预实验里有没有观察到：**没有直接看到同名问题**。
- 我看到的是另一种做法：这边整个老项目从一开始就把输入固定在 `256x256`。
  - `train_config.json` 固定 `image_size=[256,256]`
  - `infer_service.py`、`faiss_inference.py` 也都默认 `256x256`
- 我的判断：
  - 旧版这条线没有像 RU-001 那样把 `modelInputW/H` 当成可调杠杆在文档里反复讨论。
  - 但本质上他们是在用“固定输入尺寸”规避同类问题，而不是把它开放出来。

### 金矿 2：KNN 分块搜索是瓶颈
- 旧版预实验里有没有观察到：**有，而且证据非常强**。
- 证据：
  - `RECEIPT.md`：直接写“推理时间 ~5500ms，主要耗时在 KNN 搜索”
  - `TRANSFER_GPU_OPTIMIZATION_STATUS.md`：把单线程、OpenMP、FAISS 三阶段都列出来
  - `src/faiss_inference.py` 一开头就写“使用 FAISS-CPU 加速 KNN 搜索”
- 是否有人踩过坑：**踩得很深**。
  - 先被 5600ms 卡住
  - 再上 OpenMP，还是 1427ms
  - 最后才靠 FAISS 压到 83.3ms
- 是否有数据佐证：**有完整数字链条**
  - 5600 → 1427 → 258.1 / 83.3 ms

### 金矿 3：阈值被平方 `pow(thresh, 2.0)`
- 旧版预实验里有没有观察到：**没有看到直接证据**。
- 我在 `H:\RevUnsup-main` 里看到的阈值问题是：
  - `pixel_threshold=NaN`
  - 训练阈值和部署阈值需要重新校准
  - 归一化公式特殊，不是普通 min-max
- 结论：
  - 我不能证明这台机器上的旧版预实验踩过“阈值被平方”这个坑。
  - 目前只敢说：这边确实有阈值坑，但不是同一个坑，至少证据里没看到 `pow(thresh, 2.0)`。

## 8. Top 3 避坑

避坑 #1: 不要再把“暴力 KNN”当成能上线的默认方案
  - 当时试了什么：单线程 CPU 暴力搜索，后来再补 OpenMP
  - 为什么不行：5600ms 到 1427ms 都太慢，KNN 会把整条链拖死
  - 如果重复会浪费多少时间：至少会浪费一整轮部署优化，按文档看是 1 到 2 天量级

避坑 #2: 不要假设 checkpoint 里的阈值天然可部署
  - 当时试了什么：直接从 checkpoint 抽 thresholds.json 给下游用
  - 为什么不行：`pixel_threshold=NaN`，JSON 还会不合法，直接阻塞下游
  - 如果重复会浪费多少时间：至少半天到一天，因为会把导出、推理、校准都卡住

避坑 #3: 不要在这台 Windows 机器上死磕 FAISS-GPU
  - 当时试了什么：conda 安装、pip 装 NVIDIA 版本、拉预编译包
  - 为什么不行：超时、无 wheel、下载失败，平台本身不顺手
  - 如果重复会浪费多少时间：2 到 4 小时起步，而且不一定有结果

## 结论

这台 2080 上留下的最值钱经验，不是“某个模型已经神准”，而是三件事：
- 先用极小样本把流程打通，再换真实验证集看真问题。
- 阈值和部署尺度一定要单独校准，不能迷信训练产物。
- KNN 搜索才是部署瓶颈，性能优化要优先打这里。

如果羊爸爸后面要开 OM-002，我建议优先深挖两块：
- `deploy/cpp_inference/patchcore_engine.cpp` 的性能演化细节
- acceptance 集里那几张重叠样本为什么会误报/漏报
