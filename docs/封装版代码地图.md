## 4️⃣ 输出格式？

### resultPicOut
- **8位灰度图**（CV_8UC1），大小 = modelInputW × modelInputH
- **不是热力图**，是**二值化 mask**：
  - 像素值 255 = 缺陷区域
  - 像素值 0 = 正常区域
- 阈值由 `halconRegionThreshMax` 控制，DLL 内部做：
```cpp
anomaly_map.masked_fill_(anomaly_map < halconRegionThreshMax, 0);
anomaly_map.masked_fill_(anomaly_map >= halconRegionThreshMax, 255);
```

### 函数返回值
- `1` = 有缺陷
- `0` = 无缺陷（此时 resultPicOut 全黑，调用方可跳过显示加速处理）

## 8️⃣ ⭐ 最可疑/最值得深挖的地方

### 🔴 最可疑：封装版和交付版的接口签名不一致

`接口说明.h`（源码版）的 `anomalyDetMatin` 有 **9 个参数**：
```cpp
int anomalyDetMatin(int index, cv::Mat& Matin, float halconRegionThreshMax,
    const int modelInputW, const int modelInputH,
    cv::Mat& resultPicOut, int useCuda, int overlap=64, int cutSize=-1)
```

而甲方交给客户的 `说明.docx`（交付版）只有 **4 个参数**：
```cpp
int anomalyDetMatin(int index, cv::Mat& testImage, float anomalyThresh, cv::Mat& resultImage)
```

**这意味着交付版是重新封装过的简化接口**，把 `modelInputW/H`、`useCuda`、`overlap`、`cutSize` 这些参数在内部固定了。这些被隐藏的参数很可能就是**推理时间的调优杠杆**。

### 🟠 其他可疑点

1. **`demo.cpp` 出现在两个位置**：`train/demo.cpp`（43KB，可读）和 `infer/demo.cpp`（25KB，加密）。大小差异大，可能是两个不同版本，哪个是真的？

2. **Cython 编译的 `.pyd` 文件**：`torch_model.cp38-win_amd64.pyd`（250KB）和 `torch_model.py`（44KB）同时存在。`.pyd` 是编译后的，`.py` 是源码。它们内容一致吗？如果不一致，`train.exe` 到底用的哪个？

3. **`maxCutModel = 400` 和 400×400 的 `cut_index_array`**：这是一个超大的静态数组，每个 DLL 实例占用大量栈空间。如果同时加载多个模型可能导致栈溢出。

4. **模型初始化时加载整个 Memory Bank 到内存**：`MB2CPUORGPU` 函数一次性 `pickle_load` 整个 `.mb` 文件，如果特征库很大会是性能瓶颈。

5. **PaddleSeg 的模型文件被故意改名**：`note` 文件明确说"为了防止信息泄露，删除 `model.pdiparams.info`，`model.pdiparams` 改成 `params`，`model.pdmodel` 改成 `model`"——说明甲方有意混淆。

6. **`layerchoose` bug？**：`train.py` 第143行，`layerchoose[2]==1` 时追加的是 `'layer2'` 而不是 `'layer4'`，这看起来像复制粘贴错误。
