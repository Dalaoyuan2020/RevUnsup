# RU-010 L0' — FP16 预处理 Bug 诊断报告

> 2026-04-15 夜 · DogWind 执行

---

## 🎯 目标

验证/证伪 Napoleon 的假设：客户预处理顺序 (uint8 → kHalf → 归一化) 导致 FP16 精度损失 → backbone 输出偏移 → KNN 全误报。

## 🔬 方法

独立诊断程序 `diag_main.cpp`，同一张图走两条路径：

- **Path A (客户原版)**: `from_blob(kU8) → GPU → kHalf → permute → div(255) → normalize`
- **Path B (推荐顺序)**: `from_blob(kU8) → GPU → kFloat → permute → div(255) → normalize`

两条路径最终都转 FP32 对比。然后各自送入 backbone (model.ckpt) 对比输出。

## 🎯 结论

### FP16 预处理 bug: **存在，但不严重**

1. **输入层**: max_diff = 0.003，在 "minor" 区间 (0.001-0.01)
2. **Backbone层**: max_diff = 0.05，在 "bug confirmed" 区间 (> 0.01)
3. **放大效应**: 预处理误差经 backbone 放大 **19-45 倍**
4. **mean 级别**: backbone mean_diff 仅 0.0001-0.0002，非常小

### 对 "5/5 全误报" 的影响评估

FP16 预处理误差 **不太可能是 5/5 全 ret=1 的唯一原因**：
- backbone mean relative error 仅 0.2-0.5%，KNN 距离受此影响有限
- 更可能的解释：测试图本身确实包含可检测异常，或阈值设置 (thresh=0.1, 内部 pow(0.1,2)=0.01) 太低

### 建议

| 选项 | 说明 | 优先级 |
|------|------|--------|
| **L0.5 修预处理顺序** | 改 4-5 行：归一化后再 kHalf。消除 0.05 backbone 偏差 | 中 (做了更好，但不是致命) |
| **先调查 threshold** | 用不同 thresh 跑一遍看 ret 值变化 | 高 (排除阈值问题) |
| **正/负样本对照** | 找已知正常图来测试，确认是否也 ret=1 | 高 (排除测试图偏差) |
