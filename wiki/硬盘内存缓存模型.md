# 硬盘 / 内存 / 缓存 三层存储模型

## 一句话总结
**Harness 编排的最高境界 = 建立【硬盘 → 内存 → 缓存】的有效读取链, 不是把所有知识塞进上下文。**

## 来源
Napoleon 在 2026-04-09 提出, 用计算机存储层次类比 Harness 的知识流动。

## 三层定义

```
┌──────────────────────────────────────────────┐
│ 💾 硬盘 (Raw sources, immutable)              │
│   • 位置: docs/, .persona/, configs/, 任务输出│
│   • 特点: 不变, 大量, 不常访问                 │
│   • 例子: 封装版代码地图, 预实验经验清单, 编排路线│
├──────────────────────────────────────────────┤
│ 🧠 内存 (Wiki nodes, LLM 维护)                │
│   • 位置: wiki/                               │
│   • 特点: 双向链接, 简短, 高频访问             │
│   • 例子: 五层架构, KNN瓶颈, DogWind          │
├──────────────────────────────────────────────┤
│ ⚡ 缓存 (LLM 上下文)                          │
│   • 位置: 当前对话窗口                         │
│   • 特点: 最快, 最小, 任务完成后释放           │
│   • 例子: Agent 启动时被注入的 3-5 个节点      │
└──────────────────────────────────────────────┘
```

## Napoleon 的原话

> "知识是硬盘, 然后缓冲是关系是缓冲, 然后最后执行你的上下文是缓存还是内存"
> "你的上下文它是缓存, 然后内存就是那个 wiki, 然后我们的知识库它是硬盘"
> "这套有效的知识读取的方法就是影响我们最有效地去编排"
> "现在你脑子很混乱, 导致我编排它效果就很差"

## 编排的最高境界

**不是**: 把硬盘的所有东西都塞进缓存 (上下文)。
**而是**: Harness 能根据当前问题, 从硬盘 → 加载到内存 → 提取到缓存。

具体动作:
1. **L2 编排者** 接到一个新问题 (例如: "下一个杠杆试什么")
2. 先看【内存层】(wiki/index.md), 找出 3-5 个相关节点
3. 把这些节点【加载到缓存】(注入对话)
4. 基于这些节点决策, **不是从硬盘原文决策**
5. 决策后, 把新知识【ingest 回内存】(写新 wiki 节点 + 更新 log)

## 当前的问题 (Claude 的反思)

Claude 之前一直把所有东西都塞进缓存:
- 写一堆 docs/*.md 然后试图全部记住
- 没有【内存层】, 直接硬盘 ↔ 缓存
- 结果: 上下文撑爆, 编排效果差, Napoleon 觉得"脑子混乱"

正确做法:
- wiki/ 目录就是【内存层】, 严格控制在 100-200 个节点以内, 每个 50-200 行
- 大量的细节 (代码地图详情, 经验清单原文) 留在硬盘 (docs/), 不进内存
- 当前对话只 ingest 当前需要的 3-5 个节点, 不超过

## 设计原则

1. **内存层一定要小** — 不超过几十个核心节点, 太多就 LINT 合并
2. **硬盘层不要复制到内存** — 只在内存写【提炼】, 用 [[硬盘文件路径]] 引用
3. **缓存只装当前必要** — 进对话的节点不超过 5 个
4. **每个节点必须双向链接** — 形成网络, 不是树
