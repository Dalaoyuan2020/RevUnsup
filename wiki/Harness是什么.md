# Harness 是什么

## 一句话总结
**Harness 不是项目管理工具, 是让代码能自我迭代收敛的自动化框架。**

## 正文

Napoleon (羊爸爸) 反复强调过两次:
1. "我已经能本地让它跑了, 我想要的是【一套自动化框架去完善它】, 不是看到问题→定制→锤"
2. "你不跑起来的话, 根本不知道时间到底耗在什么地方"

Claude 之前一直把 Harness 当成项目管理来用 (写一个个 RU-001/RU-002/OM-001 的任务清单), 这是错的。

正确的 Harness 是:
```
跑代码 → 看哪里慢/不准 → 自动找瓶颈 → 选下一个杠杆 → 跑对照 → 收敛
   ↑___________________________________________________↓
```

不需要人手写任务, 框架自己跑完整循环。人类只在两个地方介入:
1. 设定【目标和边界】 (希望多快/多准)
2. 当框架卡住时做战略决策

这跟传统软件工程的 "PM 写需求 → 程序员写代码" 完全不同。

## 反例: 我之前的错误

```
人类: "我要优化推理时间"
Claude: 写 RU-001 (代码地图) → 喂 DogWind
        写 RU-002 (跑 baseline) → 喂 DogWind
        写 RU-003 (阈值校准) → 喂 DogWind
        写 RU-004 (杠杆 1) → 喂 DogWind
        ...
```

这是【人类驱动】, 不是 Harness。Claude 在做项目经理。

## 正例: Harness 的样子

```
人类: "我要优化推理时间, 目标 < 100ms, 误报率 < 5%"
Harness:
  循环 1: 跑 baseline → 测时间 = X, 准度 = Y
  循环 2: 自动选最有可能提速的杠杆 → 跑对照 → 测时间 = X', 准度 = Y'
  循环 3: 比较 → 保留好的, 丢弃差的
  循环 N: 一直跑到 X < 100 且 Y > 95% → 报告人类
```

人类只看【是否收敛】, 不看每个杠杆。

## 跟 [[5Q+1L Harness]] 的关系

阶段二的 5Q+1L Harness 已经把 Harness 的【最小约束】写出来了:
- Q1 做什么 (任务定义)
- Q2 看到什么 (env_check)
- Q3 记得什么 (共享 prompt)
- Q4 不能做什么 (交叉验收)
- Q5 怎么知道对了 (test_agent)
- L 灯塔 (备选方案)

这 6 个加起来就是【一个能自我迭代的框架】的最小骨架。
