# 五层架构

## 一句话总结
**Napoleon 定义的项目分层: L1 执行 → L2 编排 → L3 收敛 → L4 通讯 → L5 协议封装, 每往上一层涌现一种新能力。**

## 正文

### L1 Worker (执行)
- **干什么**: 真正跑代码, 解决现实问题
- **核心目标**: 解决【推理时间过长】问题
- **三个子目标**:
  - (a) 解决现实问题
  - (b) 性能调优 (定位耗时 → 参考经验 → 优化)
  - (c) 找耗时环节 + 针对性处理
- **当前状态**: 🔴 没人真正跑过代码

### L2 Master/主脑 (编排)
- **干什么**: 整合 Worker 输出, 找最终方案 (或几个备选)
- **当前状态**: ✅ 文档齐全, 缺"立刻能跑"的指令
- **谁**: [[Napoleon]] (Mac mini 上的 Claude + 羊爸爸)

### L3 Evaluation (收敛)
- **干什么**: 确保方案可靠, 检查代码问题
- **核心**: 【收敛性】
  - 横向收敛: 多 Agent 给的方案差异度小才算收敛
  - 纵向收敛: 一个 Agent 跑 N 次结果稳定才算可信
- **当前状态**: 📋 规范有, 没数据可评

### L4 通讯 (Agent 互相说话)
- **干什么**: 让多 Agent 在【同一个世界】里工作 (env/data/规则一致)
- **没有 L4 的后果**: 单 Agent 还能跑, 但没有协作的【涌现效应】
- **L4 的最小 4 支柱**:
  - 可见 (Q2 看到什么) — env_check.py
  - 可达 (Q3 记得什么) — 共享 prompt + lighthouse.yaml
  - 可信 (Q4 不能做什么) — 交叉验收 + [[A2A交接协议]]
  - 可标 (Q5 怎么知道对了) — test_agent*.py
- **当前状态**: ✅ A2A 协议有, 但还没被 Agent 实践过

### L5 协议封装 + 操作封装
- **干什么**: 让外界 (人类/外部系统) 能可控地操纵整个 Harness
- **核心**: 让一切【可控可外接】
- **当前状态**: 🟡 [[采用LLM_Wiki模式]] 是 L5 的一部分; "操作封装"具体定义待 Napoleon 澄清
- **当前优先级**: ⏸️ 暂缓 (优先 L1+L2+L3)

## 涌现链 (重要)

```
L1 (单 Agent 执行)
  ↓
L2 (编排能力)
  ↓
L3 (收敛性 — 让结果可信)
  ↓
L4 (多 Agent 涌现 — 1+1>2)
  ↓
L5 (外界可控)
```

每往上一层, 涌现一种新能力。少了 L4 也能跑, 但只是【单 Agent 重复】, 不会有 Agent Team 的涌现。

## 当前的根本症结

链子从 L1 断了:
```
L1 没数据 → L3 无评估对象 → L2 无决策依据 → L4 无消息可传 → L5 无操作可封装
```

打破僵局的办法: 让 L1 真正跑起来, 哪怕只跑出【一个时间数字】。
