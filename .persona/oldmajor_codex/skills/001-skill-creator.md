# Skill Creator (元技能)

> 改编自 Claude Code 内置的 skill-creator skill，精简到 DogWind 实际需要的最小集。
> 完整版：~/.claude/plugins/marketplaces/claude-plugins-official/plugins/skill-creator/skills/skill-creator/SKILL.md

## 这个技能解决什么问题

DogWind 在跑任务时总会遇到【我没有这个能力】的情况。
没有这个技能：DogWind 卡住，等人类来教。
有这个技能：DogWind 自己造一个新技能，继续干活。

## 什么时候用

触发条件（任意一个）：
- 我接到一个任务，但不知道怎么开始 → 可能缺技能
- 我在跑任务时反复重复某个流程 → 这个流程应该被沉淀为技能
- 我看到某个工具/库/API但不知道怎么用 → 缺一个'怎么用 X'的技能
- 我做完了一个非平凡的工作 → 把流程总结为可复用的技能

## 创造一个新技能的流程

### 第 1 步：明确意图

在动手写技能之前，先回答 3 个问题：
1. 这个技能让我能做什么？(一句话)
2. 什么时候会触发它？(用户说什么 / 我处于什么场景)
3. 完成后产出什么？(一个文件 / 一段输出 / 一个结论)

如果这 3 个问题答不清楚，先停下来想，不要急着写。

### 第 2 步：选编号和命名

- 看 `skills/` 目录现有的最大编号
- 新技能编号 = 最大编号 + 1
- 命名规则：`<NNN>-<技能名>.md`，技能名用 kebab-case
- 示例：当前最大是 001，新技能就是 `002-pdf-extractor.md`

### 第 3 步：写 SKILL.md（精简模板）

用以下模板新建 `.persona/skills/00X-名字.md`：

```markdown
---
name: 技能短名 (英文，kebab-case)
description: 一句话说明这个技能做什么 + 什么时候用。重要：写得"主动一点"，让我下次扫到这条时容易触发。
status: 🟡 草稿 / 🟢 已验证 / ✅ 稳定
priority: P0/P1/P2 (这个技能多重要)
---

# 技能名

## 这个技能解决什么问题
<2-3 句话描述。说清楚没有它会怎样、有了它会怎样>

## 什么时候用
<触发条件，列出 2-4 个具体场景>

## 怎么用 (步骤)

### 第 1 步：...
<具体动作>

### 第 2 步：...
<具体动作>

## 输入 / 输出
- 输入：<这个技能需要什么>
- 输出：<这个技能产出什么>

## 示例
<至少 1 个具体例子>

## 注意事项
- <边界情况>
- <已知限制>

## 状态
- 创建日期：YYYY-MM-DD
- 验证日期：YYYY-MM-DD (跑过一次没出问题就标这个)
- 引用次数：N (每次用一次就 +1)
```

### 第 4 步：先用再优化

写完草稿后，立刻用它跑当前任务。
- 如果跑通了 → 把 status 改成 🟢 已验证，记录一次引用
- 如果跑不通 → 不要立刻删，把失败原因写进"注意事项"，调整后再跑
- 如果用了 3 次都没问题 → 升到 ✅ 稳定

### 第 5 步：留痕

每次用完技能后：
1. 在技能文件的【状态】区更新引用次数
2. 如果遇到了新的边界情况，加到【注意事项】
3. 如果发现技能描述不够好（自己经常忘记触发它），优化 description 字段

## 命名规则汇总

| 编号区间 | 用途 |
|---|---|
| 000 | 示范技能（教学用） |
| 001 | 元技能（这个） |
| 002-099 | 通用技能 |
| 100-199 | RevUnsup 专用技能 |
| 200+ | 实验性技能 |

## 反过度工程化原则

写技能时遵守：

1. **一个技能只解决一个问题** — 不要造瑞士军刀
2. **能拆就拆** — 一个技能能拆成 2 个就拆
3. **大白话写** — 不要用文档腔
4. **没人用就删** — 创建 30 天没引用过的技能可以归档
5. **不为假想用例造技能** — 真的遇到了再造

## 不要做的事

- ❌ 不要为'好玩'造技能，必须解决真实问题
- ❌ 不要把技能写得太通用（容易触发不准）
- ❌ 不要把技能写得太具体（无法复用）
- ❌ 不要复制别人的技能，要从自己的需求出发
- ❌ 不要造一个'造技能的技能的技能'（已经有 001 了）

## 自检清单

每次创造一个新技能后，自检：
- [ ] 编号正确（连续，没和已有冲突）
- [ ] 文件名是 kebab-case
- [ ] frontmatter 三个字段都填了 (name, description, status)
- [ ] 'description' 字段足够'主动'，让自己下次能想起来
- [ ] 至少有 1 个示例
- [ ] 'status' 反映真实状态（不要直接写'已验证'）
- [ ] 已经在当前任务中用过一次（先用再升级）
